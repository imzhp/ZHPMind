# 软件 3.0（Software 3.0）

软件 3.0 是由 [[andrej-karpathy]] 提出的软件开发范式框架，将大语言模型（LLM）定义为新型可编程计算机：以"提示（Prompt）"为编程语言，以"上下文窗口（Context Window）"为控制杆，以 LLM 为解释器（Interpreter）。这一范式不是对已有开发方式的加速，而是催生了一类此前不存在的信息处理可能性。

## 三代范式对比

| 范式 | 编程方式 | 程序的载体 |
|------|---------|-----------|
| **软件 1.0** | 手写代码（显式规则、确定性逻辑） | 可执行程序 |
| **软件 2.0** | 设计数据集 + 目标函数，训练神经网络 | 学习到的权重（Learned Weights） |
| **软件 3.0** | 构建提示与上下文（Prompting） | LLM 作为解释器，在上下文中执行计算 |

软件 2.0 的革命在于：程序员不再写规则，而是"通过安排数据集和神经网络架构来编程"。软件 3.0 更进一步——当 GPT 等模型在足够大规模的多任务互联网数据上训练后，因为需要同时完成数据集中隐含的所有任务，它们成为了某种意义上的"可编程计算机"。

## 软件 3.0 下的编程模型

- **Prompt = 代码**：给 Agent 的文本指令就是程序本体，而不是配置项
- **上下文窗口 = 运行时状态**：你对 LLM 解释器的唯一杠杆
- **Agent = 带智能的执行环境**：不只是执行指令，还能理解环境、自行调试

Karpathy 举例：OpenClaw 的安装方式不再是执行 Shell 脚本（软件 1.0），而是"把一段文本粘贴给你的 Agent，它会理解你的环境并自行调试安装"。Shell 脚本随着平台增多会膨胀成极其复杂的脚本；Agent 则带着自己的智能，直接适应环境。

## Menu Gen：新旧范式的对比案例

这是 Karpathy 用来展示范式差距的核心案例：

**软件 1.0/2.0 方式**（Karpathy 自己构建的 Menu Gen）：
1. 拍菜单照片上传
2. OCR 识别菜单中的所有菜品名称
3. 调用图像生成 API，为每道菜生成图片
4. 渲染 UI 展示带图的菜单
5. 部署到 Vercel

**软件 3.0 方式**（后来有人展示给他看的）：
1. 拍照
2. 发给 Gemini，说"用 Nanabanana 把菜品图片叠加到菜单上"
3. 直接输出：带菜品图片叠加的菜单原图

Karpathy 的反应："这让我意识到我的 Menu Gen app 是多余的，它本不应该存在。"

## 全新的可能性空间

软件 3.0 最重要的含义不是"已有事物变快了"，而是**以前不在可能性空间里的事物现在变得可能**：

- **以前**：代码处理结构化数据；没有代码能"基于一堆文档自动创建知识库"
- **现在**：LLM 可以摄取非结构化信息（文档、图片、音频），重新组织、重新投影，产出新形式的知识——这是全新的信息处理类型

Karpathy 认为这比"已有工作流程加速"更令人兴奋：新的可能性空间，等待探索。

## 对应用设计者的含义

软件 3.0 范式要求从根本上重新思考应用设计：
- 不要问"如何用 AI 加速已有的 app"
- 要问"这件事是否根本不需要 app，直接给 LLM 就能完成？"
- 未来的"应用"可能就是一段精心设计的 Prompt，输入是原始数据，输出是结果，中间层消失

见 [[agentic-engineering]] 和 [[vibe-coding]] 了解这一范式在工程实践中的两种表现形式。
